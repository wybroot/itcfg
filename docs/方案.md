# ITCFG 配置中台 + 部署包编排系统 — 项目方案

> 版本: v1.0 | 日期: 2026-07-15 | 状态: 方案设计

---

## 一、项目背景与痛点

### 1.1 业务现状

当前业务涉及 **14+ 组件** 的部署运维：

| 类别 | 组件 |
|------|------|
| 应用服务 | Java 应用 |
| Web 服务器 | Nginx |
| 数据库 | MySQL / PostgreSQL、MongoDB |
| 缓存 | Redis |
| 消息队列 | Kafka |
| 对象存储 | MinIO |
| 协调服务 | Etcd |
| 搜索引擎 | Elasticsearch (IK 分词) |
| 办公套件 | OnlyOffice、Collabora Code (25.04.9.4.1) |
| 文件服务 | File Viewer、FileCodeBox |

### 1.2 核心痛点

1. **配置繁琐**：每个组件都有独立的配置文件（nginx.conf、application.yml、redis.conf 等），格式各异，每次部署需逐个修改
2. **客户差异大**：不同客户现场的网络环境、IP 地址、端口、账号密码均不同，无法复用
3. **人员要求高**：部署人员需熟悉所有组件的配置语法，技术门槛过高
4. **版本混乱**：各组件版本无统一管理，现场部署时不知道应该用哪个版本
5. **无迹可循**：每次部署靠人工记录，部署了什么版本、改了什么配置，事后难追溯
6. **离线场景**：客户现场往往网络隔离，无法在线拉取镜像和配置

### 1.3 期望目标

> 运维/开发人员在配置中台按客户环境录入配置 → 一键导出完整部署包 → 现场人员拿到部署包后执行一条命令完成部署，**零技术门槛、零出错、全可追溯**。

### 2.2 系统组成

| 系统 | 职责 | 技术选型 |
|------|------|----------|
| **配置中台 (Web)** | 客户管理、模板管理、配置录入、版本管理、部署记录 | Go (Gin) + Vue 3 + PostgreSQL |
| **制品仓库** | 存储 Docker 镜像、Jar 包等制品 | Harbor (CNCF 毕业项目) |
| **部署包编排服务** | 按客户+环境组装完整部署包并导出 | 集成在配置中台后端 |
| **Config Agent** | 客户端工具，解包、部署、启动 | Go，编译为单一二进制文件 |

### 2.3 设计原则

- **配置与制品分离**：配置中台只存文本配置，不存镜像/包，各司其职
- **在线与离线并重**：在线场景 API 下发，离线场景导出打包
- **模板化**：所有组件配置均模板化，新增组件只需添加模板，不改代码
- **零门槛部署**：现场部署人员只需执行一条命令
- **全链路可追溯**：每次部署记录版本、配置、人员、时间

## 三、配置中台详细设计

### 3.1 核心模块

#### 3.1.1 客户管理

```
客户 → 环境 → 组件 → 配置值
```

- 每个客户可创建多个环境（生产、测试、灾备）
- 每个环境可选择需要的组件
- 支持客户配置的克隆（客户A → 客户B 快速创建）

#### 3.1.2 组件模板管理

每个组件定义一套标准模板，结构如下：

```
templates/
├── nginx/
│   ├── manifest.yaml          # 组件元信息
│   ├── variables.yaml         # 变量定义
│   └── files/
│       ├── nginx.conf.tmpl
│       └── conf.d/
│           └── default.conf.tmpl
├── java-app/
│   ├── manifest.yaml
│   ├── variables.yaml
│   └── files/
│       ├── application.yml.tmpl
│       ├── logback.xml.tmpl
│       └── jvm.options.tmpl
├── postgresql/
│   ├── manifest.yaml
│   ├── variables.yaml
│   └── files/
│       ├── postgresql.conf.tmpl
│       └── pg_hba.conf.tmpl
└── ... (其他组件)
```

#### 3.1.3 配置录入界面

根据 `variables.yaml` 自动生成表单，按分组展示，支持类型校验。

#### 3.1.4 配置渲染引擎

使用 Go 的 `text/template` 渲染模板，变量替换生成最终配置文件。

#### 3.1.5 版本关联

配置中台不存制品本身，只存版本号关联（组件名、制品类型、版本号、Harbor 地址）。

#### 3.1.6 部署记录

每次部署记录版本标签、配置快照(JSONB)、制品版本快照、校验和、部署人员、状态。

## 四、Config Agent 详细设计

### 4.1 技术选型

- **语言：** Go 1.22+
- **编译目标：** 单一二进制文件，零依赖
- **支持平台：** Linux amd64 / arm64

### 4.2 命令设计

```bash
# 离线部署模式
config-agent import --package ./deploy-package.tar.gz
config-agent deploy

# 工具命令
config-agent validate     # 校验
config-agent status       # 查看状态
config-agent rollback     # 回滚
config-agent version      # 版本
```

### 4.3 部署包结构

```
customer-a-prod-v1.2.3.tar.gz
├── metadata.json          # 元信息
├── images/                # Docker 镜像 tar
├── configs/               # 渲染好的配置文件
├── docker-compose.yml     # 编排文件
├── install.sh             # 一键安装
├── start.sh / stop.sh     # 启停脚本
└── uninstall.sh           # 卸载
```

## 六、技术选型

| 组件 | 技术 | 原因 |
|------|------|------|
| 配置中台后端 | Go (Gin/GORM) | 高性能、单二进制、与 Agent 同语言 |
| 配置中台前端 | Vue 3 + Element Plus | 快速开发、生态成熟 |
| 配置中台数据库 | PostgreSQL 15+ | JSONB 支持、成熟稳定 |
| 制品仓库 | Harbor | CNCF 毕业、企业级 |
| Config Agent | Go | 编译为单一二进制、零依赖、跨平台 |
| 模板引擎 | Go text/template | 标准库、无需额外依赖 |
| 部署方式 | Docker Compose | 客户现场统一用容器化部署 |

## 七、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| Docker 镜像体积大 | 传输慢 | 分卷压缩、USB 3.0 移动硬盘 |
| 客户现场环境差异大 | 部署失败 | Agent 前置校验 |
| 组件版本兼容性 | 服务异常 | 版本兼容性矩阵 |
| 配置中台单点故障 | 无法导出 | 数据库备份、部署包离线可用 |
| 人员流动 | 维护困难 | 模板化、文档完善 |
