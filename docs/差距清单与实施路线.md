# ITCFG 项目差距清单与真实可用实施路线

> 编写日期：2026-07-17  
> 目的：基于当前代码状态，对照真实诉求，明确哪些能力已具备、哪些只是雏形、哪些必须优先补齐，作为后续开发参考。

---

## 一、结论

当前项目**还不是一个能满足真实交付需求的完整项目**。

它目前更接近一个“配置中台 + 部署系统”的 MVP 样板：

- 中台页面和基础 API 已经有雏形；
- 客户、环境、组件、配置、版本、部署记录等模型已经搭起来；
- 模板渲染和配置导出已经初步存在；
- Agent 有 CLI 框架，但真实部署能力还不完整。

但距离真实目标：

```text
配置中台录入客户环境配置
        ↓
选择组件和制品版本
        ↓
一键生成完整离线部署包
        ↓
客户现场拿包后执行一条命令
        ↓
自动导入镜像、写配置、启动服务、健康检查、记录状态
```

仍有明显差距。

当前不能继续只堆 CRUD 页面，应该优先打通“完整离线部署闭环”。

---

## 二、真实诉求重新定义

项目真正要解决的不是“有一个配置管理页面”，而是解决企业现场交付的几个核心痛点。

### 2.1 配置复杂

现状痛点：

- 每个组件配置格式不同；
- Nginx、Java、PostgreSQL、Redis、MinIO、Kafka、OnlyOffice 等都有独立配置；
- 人工改配置容易错；
- 现场部署人员不一定懂每个组件。

真实目标：

- 中台通过表单录入配置；
- 所有配置由模板自动渲染；
- 配置项有类型、默认值、必填、校验规则；
- 现场人员不直接修改配置文件。

### 2.2 客户差异大

现状痛点：

- 不同客户 IP、端口、域名、账号、密码、存储路径不同；
- 同一客户还有生产、测试、灾备等多环境；
- 配置无法简单复制。

真实目标：

- 客户、环境、组件、配置值要有清晰层级；
- 支持复制已有客户/环境配置；
- 支持按环境导出独立部署包；
- 环境配置必须能追溯和回滚。

### 2.3 制品版本混乱

现状痛点：

- 现场不知道每个组件应该部署哪个镜像或 Jar；
- 镜像、配置、部署记录之间没有强关联；
- 离线场景无法现场拉取镜像。

真实目标：

- 中台记录每个环境下每个组件的制品版本；
- 导出部署包时按版本拉取或收集镜像；
- 部署包内包含 `images/`、`image-list.txt`、`docker-compose.yml`；
- 部署记录里保存配置快照和制品快照。

### 2.4 离线部署

现状痛点：

- 客户现场常常网络隔离；
- 不能依赖现场访问公网、Harbor 或配置中台；
- 现场人员希望拿包即部署。

真实目标：

- 中台导出完整 `.tar.gz` 部署包；
- 部署包包含运行所需的一切：配置、镜像、compose、脚本、元信息；
- Agent 在离线环境中完成导入、校验、部署、状态查看、回滚。

### 2.5 零门槛部署

现状痛点：

- 现场人员不一定懂 Docker、配置语法、端口依赖；
- 出错后难定位；
- 失败后难恢复。

真实目标：

```bash
config-agent install customer-prod-v1.0.0.tar.gz
```

或者：

```bash
config-agent import --package customer-prod-v1.0.0.tar.gz
config-agent validate --env --package
config-agent deploy
config-agent status
```

Agent 需要做到：

- 自动解包；
- 自动校验；
- 自动导入镜像；
- 自动写配置；
- 自动启动服务；
- 自动健康检查；
- 自动保存部署记录；
- 失败可查看日志；
- 支持回滚。

## 四、当前关键差距

### 4.1 部署包不是完整部署包

当前导出更接近“配置包”，不是完整离线部署包。

真实部署包应该包含：

```text
customer-prod-v1.0.0.tar.gz
├── metadata.json
├── manifest.json
├── checksums.txt
├── images/
│   ├── nginx_1.25.tar
│   ├── app_1.2.3.tar
│   └── postgres_17.tar
├── image-list.txt
├── configs/
│   ├── nginx/
│   ├── java-app/
│   └── postgresql/
├── docker-compose.yml
├── scripts/
│   ├── install.sh
│   ├── start.sh
│   ├── stop.sh
│   ├── uninstall.sh
│   └── healthcheck.sh
└── agent/
    └── config-agent
```

当前缺少：

- 镜像打包；
- compose 生成；
- 脚本写入；
- 校验文件；
- Agent 打包；
- 完整 manifest；
- 制品版本快照。

### 4.2 Agent 不具备真实部署能力

当前 Agent 存在明显问题：

- 启动服务方法存在无限递归风险；
- Docker load 逻辑没有真正执行；
- docker-compose up/down 逻辑没有真正执行；
- 配置写入目标路径设计不够清晰；
- rollback 只是初步逻辑；
- 缺少部署日志；
- 缺少幂等能力；
- 缺少健康检查闭环。

Agent 是真实交付的核心，必须优先补齐。

### 4.3 制品版本没有形成闭环

当前已有制品版本记录字段，但只是“录入信息”。

真实闭环应该是：

```text
环境组件选择制品版本
        ↓
根据 registry_url / artifact_name / artifact_version 定位镜像
        ↓
导出时 docker pull
        ↓
导出时 docker save 到 images/
        ↓
生成 image-list.txt
        ↓
生成 docker-compose.yml
        ↓
Agent 现场 docker load
        ↓
compose 启动对应版本
```

当前缺少：

- Harbor/Registry 认证配置；
- 镜像拉取；
- 镜像保存；
- 镜像文件和制品版本映射；
- compose image 字段生成；
- 制品快照写入部署记录。

### 4.4 组件模板不足

方案目标是 14+ 组件，当前实际模板覆盖不足。

当前至少需要补齐这些组件：

1. Nginx
2. Java App
3. PostgreSQL
4. MySQL
5. Redis
6. MinIO
7. Kafka
8. MongoDB
9. Etcd
10. Elasticsearch
11. OnlyOffice
12. Collabora Code
13. File Viewer
14. FileCodeBox

但建议不要一口气补 14 个，应先用 3 个组件打通完整闭环。

### 4.5 配置校验不足

当前配置校验更多是变量级别。

真实部署需要：

- 必填校验；
- 类型校验；
- 端口范围校验；
- IP/域名/URL 校验；
- 路径校验；
- 密码复杂度或敏感字段校验；
- 组件间依赖校验；
- 端口冲突校验；
- compose 语法校验；
- 组件配置语法校验。

例如：

- Nginx：`nginx -t`；
- PostgreSQL：配置项合法性；
- Docker Compose：`docker compose config`；
- 端口：检查宿主机是否占用。

### 4.6 追溯能力还不完整

当前有部署记录和配置版本，但真实追溯需要记录：

- 谁导出了部署包；
- 导出时配置快照；
- 导出时制品快照；
- 部署包 SHA256；
- 部署包内容 manifest；
- 谁在现场执行部署；
- 部署开始时间、结束时间；
- 部署日志；
- 健康检查结果；
- 失败原因；
- 回滚记录。

当前只具备部分基础。

## 六、建议的实施路线

## 阶段一：打通最小真实离线部署闭环

目标：只支持 3 个组件，但必须真实可用。

建议组件：

1. Nginx
2. Java App
3. PostgreSQL

### 6.1 后端需要完成

#### 6.1.1 重构部署包结构

导出包必须包含：

```text
metadata.json
manifest.json
checksums.txt
configs/
images/
docker-compose.yml
scripts/
```

#### 6.1.2 生成 docker-compose.yml

基于环境选择的组件和制品版本生成 compose。

需要解决：

- 服务名；
- 镜像名；
- 端口映射；
- volume；
- env；
- depends_on；
- healthcheck；
- restart 策略。

#### 6.1.3 写入部署脚本

生成：

- `scripts/install.sh`；
- `scripts/start.sh`；
- `scripts/stop.sh`；
- `scripts/uninstall.sh`；
- `scripts/healthcheck.sh`。

#### 6.1.4 生成 checksums.txt

对包内关键文件计算 SHA256：

- 镜像 tar；
- 配置文件；
- compose 文件；
- metadata；
- manifest。

#### 6.1.5 修正 metadata checksum 写入时机

当前应避免“打包后才把 checksum 写进 metadata，但包内 metadata 不是最终版本”的问题。

建议：

- `manifest.json` 记录包内容清单；
- `checksums.txt` 记录文件级 checksum；
- 外层下载响应 header 返回整个 tar.gz 的 checksum；
- 不强行把包自身 checksum 写入包内 metadata，避免循环依赖。

### 6.2 Agent 需要完成

#### 6.2.1 修复部署执行器

必须实现真实逻辑：

```text
import
  - 解压 tar.gz
  - 防路径穿越
  - 校验 manifest/checksums
  - 保存当前部署包

validate
  - 检查 Docker
  - 检查 docker compose
  - 检查端口
  - 检查磁盘空间
  - 检查配置文件存在
  - 检查镜像文件存在

deploy
  - 备份旧配置
  - docker load 镜像
  - 写配置到工作目录
  - docker compose config
  - docker compose up -d
  - healthcheck
  - 写本地部署状态

status
  - 显示当前版本
  - 显示服务状态
  - 显示健康检查结果

rollback
  - 停止当前服务
  - 恢复上一次部署目录
  - 重新 compose up
```

#### 6.2.2 明确现场目录结构

建议 Agent 使用固定工作目录：

```text
/opt/itcfg/
├── packages/
│   ├── customer-prod-v1.0.0/
│   └── customer-prod-v1.0.1/
├── current -> packages/customer-prod-v1.0.1
├── previous -> packages/customer-prod-v1.0.0
├── logs/
├── state.json
└── backup/
```

不要直接把配置散写到根目录。

#### 6.2.3 增加 install 命令

为了贴合真实使用体验，建议增加：

```bash
config-agent install ./customer-prod-v1.0.0.tar.gz
```

等价于：

```bash
config-agent import --package ./customer-prod-v1.0.0.tar.gz
config-agent validate --package --env
config-agent deploy
config-agent status
```

### 6.3 第一阶段验收标准

第一阶段完成后，必须满足：

```bash
# 云端导出
在 Web 中选择客户环境，导出 customer-prod-v1.0.0.tar.gz

# 现场部署
scp customer-prod-v1.0.0.tar.gz target:/tmp/
config-agent install /tmp/customer-prod-v1.0.0.tar.gz
config-agent status
```

并且：

- 能导入镜像；
- 能启动 Nginx + Java App + PostgreSQL；
- 配置来自中台；
- compose 使用指定制品版本；
- 部署失败能看到错误；
- 重复执行不会破坏现场；
- 能回滚到上一个版本。

## 阶段二：打通制品版本与 Harbor/Registry

目标：让“制品版本管理”从录字段变成真实可用。

### 6.4 后端新增能力

#### 6.4.1 Registry 配置

需要新增 Registry/Harbor 配置：

- 名称；
- 地址；
- 用户名；
- 密码或 token；
- 是否启用；
- TLS 配置；
- 项目命名空间。

#### 6.4.2 镜像拉取与保存

导出包时：

```bash
docker pull registry/app:1.2.3
docker save registry/app:1.2.3 -o images/app_1.2.3.tar
```

或者使用容器镜像库 API/SDK 完成类似能力。

#### 6.4.3 制品快照

导出时固化：

```json
{
  "component": "java-app",
  "artifact_type": "docker-image",
  "image": "registry.example.com/project/app:1.2.3",
  "saved_as": "images/app_1.2.3.tar",
  "digest": "sha256:..."
}
```

### 6.5 第二阶段验收标准

- 中台录入制品版本；
- 导出时自动收集对应镜像；
- 部署包内包含镜像 tar；
- Agent 可离线 docker load；
- compose 使用准确镜像版本；
- 部署记录能追溯制品快照。

## 阶段三：补齐核心组件模板

目标：从 3 个组件扩展到真实业务需要的组件。

建议顺序：

1. Redis
2. MinIO
3. MySQL
4. Elasticsearch
5. Kafka
6. MongoDB
7. Etcd
8. OnlyOffice
9. Collabora Code
10. File Viewer
11. FileCodeBox

每个组件模板必须包含：

```text
manifest.yaml
variables.yaml
files/*.tmpl
compose service 模板
healthcheck 定义
默认端口定义
volume 定义
依赖定义
```

### 6.6 模板标准化要求

每个组件的 `manifest.yaml` 应至少包含：

```yaml
name: nginx
display_name: Nginx
category: web-server
version: 1.0
service:
  name: nginx
  default_image: nginx:1.25
  ports:
    - name: http
      container_port: 80
      variable: nginx_port
  volumes:
    - source: ./configs/nginx/nginx.conf
      target: /etc/nginx/nginx.conf
  healthcheck:
    type: http
    url: http://127.0.0.1:{{ .nginx_port }}/health
```

`variables.yaml` 应支持：

- 类型；
- 默认值；
- 必填；
- 分组；
- 校验规则；
- 是否敏感；
- 描述；
- 选项。

## 七、建议的数据模型调整

### 7.1 EnvironmentComponent

当前模型里环境和组件的启用关系不够明确，建议增加：

```text
EnvironmentComponent
- id
- customer_env_id
- component_id
- enabled
- sort_order
- deploy_group
```

用途：明确某个环境到底部署哪些组件。

### 7.2 RegistryConfig

新增：

```text
RegistryConfig
- id
- name
- registry_url
- username
- password_encrypted
- project
- insecure_skip_verify
- is_active
```

用途：支持 Harbor/Registry 拉取镜像。

### 7.3 PackageExportRecord

新增：

```text
PackageExportRecord
- id
- customer_env_id
- version_tag
- package_name
- package_path
- package_checksum
- config_snapshot
- artifact_snapshot
- exported_by
- exported_at
- status
- error_message
```

用途：追溯每一次导出。

### 7.4 ComponentComposeSpec

可以放在模板 manifest 中，不一定入库。

## 十一、阶段一完成标志

只有满足下面条件，才算项目从“样板”进入“真实可用 MVP”：

```bash
# 中台侧
1. 创建客户
2. 创建环境
3. 启用 nginx/java-app/postgresql
4. 录入配置
5. 录入制品版本
6. 导出完整部署包

# 现场侧
1. 拷贝部署包
2. 执行 config-agent install package.tar.gz
3. 自动导入镜像
4. 自动启动服务
5. 健康检查通过
6. config-agent status 显示正确状态
7. config-agent rollback 能恢复上一版本
```

如果上述闭环没有完成，即使页面再多，也仍然不能认为项目满足真实需求。

## 十二、最终建议

后续开发应围绕一个原则：

> 先让一个真实客户环境能完整离线部署成功，再扩展组件数量和管理功能。

不建议下一步继续优先做：

- 页面美化；
- 更多 CRUD；
- 更多仪表盘；
- 复杂权限；
- 过早补齐全部 14 个组件。

建议下一步直接进入：

```text
完整部署包规范设计
        ↓
PackageExporter 重构
        ↓
Agent 真实部署能力修复
        ↓
Nginx + Java App + PostgreSQL 端到端离线部署验收
```

这才是当前项目从“半成品”走向“真实可交付”的关键路径。
